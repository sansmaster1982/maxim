# Этап 2b — Надёжное наполнение списка чатов

Дата: 2026-06-04. Цель: сделать наполнение списка чатов устойчивым к случаю, когда compact-msgpack чатов в ответе LOGIN не декодируется целиком.

## Находка из эталонного клиента

`maxclient/core/service.py._ingest_login` НЕ доверяет декодированным объектам чатов. Он:
1. байт-сканом достаёт id чатов из сырого тела LOGIN (`raw_parsers.extract_chat_ids_from_login_raw`);
2. вызывает CHAT_INFO (op 48) и наполняет названия/аватары.

Это надёжнее чистого декода: большой LOGIN-payload в compact/ref-кодировании msgpack иногда не распаковывается, а id чатов всё равно вытаскиваются по байтовым сигнатурам.

Второе наблюдение: `maxclient._parse_push` (как и наш `_parsePush`) обрабатывает только НОВЫЕ сообщения, а кадры mark/delete/reactions молча отбрасывает по «нет chatId+контента». То есть авторитетного референса для разбора push 130/142/155 нет ни в `bridge.py`, ни в `maxclient`. Угадывать имена полей нельзя — отложено (см. Этап 5).

## Сделано

- **`max_client.dart`**: при `interactive`-логине кешируется не только декодированный снэпшот, но и сырое распакованное тело (`lastLoginRaw`) — нужно для байт-скана.
- **`raw_parsers.dart`**: `extractChatIds(data)` — порт `extract_chat_ids_from_login_raw`. Две сигнатуры: `\xA2id` + (int32/int64) с маркером `DIALOG`/`CHAT`/`CHANNEL` в пределах 120 байт; и `\xA6chatId` + (int32/int64). Реализован байтовым сканом (без regex по бинарю), с дедупом и пропуском нулей.
- **`sync_repository.dart`**: двухуровневое наполнение.
  1. Богатые чаты из декода (title, превью, unread) — merge-upsert с сохранением локальных флагов pin/archive/mute.
  2. id из байт-скана, которых не было в декоде, — заглушка-строка, затем
  3. чаты без названия добираются через `ChatsRepository.refresh(ids)` (CHAT_INFO op 48, тоже merge-upsert).
- **`providers.dart`**: `SyncRepository` получает `ChatsRepository` для добора через CHAT_INFO.

## Верификация

- `flutter analyze` — чисто.
- `flutter test` — 34 кейса, новый `raw_parsers_test.dart` (3 кейса: извлечение chatId/id с маркером и пропуск «голого» id, дедуп, пустые данные).

## Отложено (нет референса либо нужен живой pcap)

- Разбор push-опкодов 130 NOTIF_MARK, 142 NOTIF_MSG_DELETE, 155 NOTIF_REACTIONS — эталонные клиенты их не парсят, имена полей неизвестны.
- Inbound typing (опкод push не подтверждён).
- DNS-over-HTTPS fallback, zstd-кадры (`cof=0xFF`) — перенесены в Этап 5.
- Приём новых сообщений (128) работает; транскрипция по запросу (op 202) работает, не обрабатывается лишь готовность-push 293.

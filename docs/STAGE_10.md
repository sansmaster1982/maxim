# Этап 10 — Корень банов номера: отсутствие keepalive (2026-06-06)

Версия `0.1.2`. Симптом: при входе через наши клиенты (`max iso`, `max new
maxim`, `max exe python`) номер блокируется, а при входе через `test5.py` —
нет. Задача: найти настоящую причину, а не гадать.

## Метод: что общего у банящих и чего нет у безопасного

Сравнили четыре клиента по login/handshake и поведению соединения.

| Признак | test5.py (НЕ банит) | max iso / max new maxim / max exe python (банят) |
| --- | --- | --- |
| userAgent в INIT | минимальный (3 поля) | max iso/new — полный (11); **max exe — минимальный (3)** |
| deviceId | случайный每 запуск | стабильный (max iso) / разный |
| LOGIN interactive | false | max exe — **false**; iso/new — true |
| авто-reconnect | **нет** | **есть у всех трёх** |
| keepalive PING | нет | **нет ни у кого** |
| активность после входа | ничего (ждёт меню) | автоматически грузят чаты/контакты |

Два вывода сразу отсекают ложные версии:

- **userAgent ни при чём.** `max exe python` шлёт ТОТ ЖЕ минимальный userAgent,
  что и `test5.py`, но банит. Значит обогащение userAgent (Этап 7) — не причина
  банов и не лечение. Поле userAgent можно не трогать.
- **deviceId ни при чём.** `test5.py` берёт новый случайный deviceId на каждый
  запуск и НЕ банит. Значит «новое устройство на номере» само по себе не банит.

Остаётся ровно один признак, общий у трёх банящих и отсутствующий у безопасного:
**авто-reconnect**. И связанный с ним: **никто не шлёт keepalive**.

## Корневая причина

Официальный клиент шлёт **PING — opcode 1** (декомпил APK,
`defpackage/ewc.java`: `new ewc("PING", 0, 1, null)`), чтобы сервер не рвал
простаивающее соединение. Наши клиенты PING не слали никогда.

Цепочка:
1. Нет PING → сервер/NAT рвёт сокет по idle за десятки секунд.
2. `_ReconnectManager` на КАЖДЫЙ успешный reconnect сбрасывает задержку в 2с.
3. Сокет снова простаивает → снова дроп → reconnect через 2с → новый INIT+LOGIN.
4. Итог: новый LOGIN каждые ~30с = шторм создания сессий на одном номере.
5. Антифрод MAX видит автоматизированный re-auth → **бан номера.**

`test5.py` не реконнектит вообще: дропнулся сокет — скрипт просто завершился,
шторма нет. Поэтому он «безопасный» — не потому, что он что-то делает правильно,
а потому, что он НЕ делает вредное (не долбит reconnect+login).

## Фикс (в `max iso`, версия 0.1.2)

- **Keepalive.** `MaxOp.ping = 1`. `Timer.periodic` шлёт PING раз в 25с, пока
  сокет жив (старт в `connect`, стоп в `_disconnect`). Соединение держится →
  сервер не рвёт по idle → reconnect почти не запускается.
- **Гашение шторма (defense in depth).** База reconnect 2с→5с, плюс джиттер
  0–1500мс. Даже при редком реальном разрыве клиент не частит и не бьёт сервер
  ровным «роботическим» залпом.

Проверка: `flutter analyze` чисто, 72 теста зелёные.

## Честная оговорка

1. **Уже забаненный номер клиентом не лечится.** Фикс предотвращает БУДУЩИЕ
   баны. Проверять надо на свежем/разбаненном номере.
2. Это сильная, обоснованная версия (PING реален, механизм шторма логичен,
   сравнение клиентов чистое), но эмпирически частоту reconnect мы не замеряли.
   Как подтвердить: в логах клиента считать события INIT/LOGIN/reconnect. До
   фикса при простое их было бы много (каждые ~30с); после — один LOGIN на
   сессию и PING раз в 25с, reconnect почти не возникает.
3. Поведенческие баны (спам, скрейпинг контактов, датацентровый/зарубежный
   VPN-IP) keepalive не отменяет — это отдельный слой (Этап 7, троттлинг op 46).
